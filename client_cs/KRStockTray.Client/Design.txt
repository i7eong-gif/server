KRStockTray - Developer Design Notes
===================================

이 문서는 KRStockTray 프로젝트의
설계 의도, 구조, 그리고 "왜 이렇게 만들었는지"를 설명하기 위한
개발자용 문서이다.

코드를 수정하거나 기능을 추가하기 전에
반드시 이 문서를 한 번 읽고 시작할 것을 권장한다.


--------------------------------------------------
1. 프로젝트 목표
--------------------------------------------------

KRStockTray는 다음을 목표로 한다.

- Windows 트레이에 상주하는 초경량 주식 모니터
- 항상 켜두어도 부담 없는 리소스 사용
- 설정 변경 시 즉각적인 UI 반영
- 서버/클라이언트 역할이 명확히 분리된 구조
- 유지보수 가능한 단순한 설계


--------------------------------------------------
2. 핵심 설계 철학
--------------------------------------------------

[1] 단일 진실 소스 (Single Source of Truth)

- 앱 전체에서 상태는 단 한 곳에서만 관리한다.
- SettingsService.Config가 유일한 상태 저장소이다.
- UI는 상태를 "보여주기만" 하고, 상태를 복제하지 않는다.


[2] 복사본 금지

- WatchList는 복사해서 사용하지 않는다.
- ObservableCollection을 그대로 바인딩한다.
- List<T> → ObservableCollection<T>로 변환하는 코드는
  반드시 제거해야 한다.


[3] 즉시 반영과 저장의 분리

- NotifyChanged()
  * UI 및 시세 재조회 트리거
  * 메모리 내 상태 변경 알림

- Save()
  * 디스크에 설정 저장
  * 절대 UI 갱신 트리거 역할을 하지 않는다


[4] WPF 바인딩을 신뢰한다

- UI 갱신을 수동으로 강제하지 않는다.
- Items.Refresh()는 사용하지 않는다.
- INotifyPropertyChanged + ObservableCollection에 맡긴다.


--------------------------------------------------
3. 전체 아키텍처 개요
--------------------------------------------------

[Stock API Server]
        ▲
        │
   QuotePoller
        │
SettingsService (Config)
        │
 ┌──────┴────────┐
 │               │
MainWindow   SettingsWindow


- SettingsService는 앱의 중심이다.
- UI는 SettingsService를 구독한다.
- Poller는 SettingsService 변경 이벤트에 반응한다.


--------------------------------------------------
4. 주요 구성 요소 상세
--------------------------------------------------

--------------------------------------------------
4.1 App.xaml.cs
--------------------------------------------------

- 애플리케이션 진입점
- 단일 인스턴스 보장 (Mutex)
- 트레이 아이콘 생성 및 관리
- MainWindow / SettingsWindow 생성 책임

중요 포인트:
- ShutdownMode는 OnExplicitShutdown 사용
- MainWindow는 즉시 Show하지 않는다
- 빈 창 깜빡임 방지를 위해 초기 Hide 처리


--------------------------------------------------
4.2 SettingsService
--------------------------------------------------

역할:
- 전체 설정 로드 / 저장
- 상태 변경 이벤트 발행
- WatchList 및 UI 상태 보관

중요 필드:
- AppConfig Config
- event OnConfigChanged

중요 메서드:
- LoadOrCreate()
- NotifyChanged()   // 즉시 반영
- Save()            // 디스크 저장


설계 주의사항:
- SettingsService는 UI를 참조하지 않는다.
- SettingsService는 Poller를 직접 호출하지 않는다.
- 모든 상태 변경은 Config를 통해 이루어진다.


--------------------------------------------------
4.3 AppConfig
--------------------------------------------------

사용자 설정 데이터 컨테이너

포함 항목:
- WatchList (ObservableCollection<WatchItem>)
- WindowOpacity
- WindowLeft
- WindowTop

주의:
- AppConfig에는 로직을 넣지 않는다.
- 순수 데이터 컨테이너 역할만 한다.


--------------------------------------------------
4.4 WatchItem
--------------------------------------------------

개별 종목을 나타내는 모델

필드:
- Market   : KR / US
- Code     : 종목 코드
- Name     : 종목명 (선택)
- IsEnabled: 활성 여부

특징:
- INotifyPropertyChanged 구현
- 속성 변경 시 UI 자동 갱신

주의:
- WatchItem은 immutable로 만들지 않는다.
- UI 편의성을 위해 mutable 구조를 유지한다.


--------------------------------------------------
4.5 QuotePoller
--------------------------------------------------

역할:
- 서버에서 주가를 주기적으로 조회
- 활성 종목(IsEnabled=true)만 조회
- 결과를 MainWindow에 전달

동작 방식:
- 백그라운드 Task 루프
- 일정 주기로 Fetch
- OnConfigChanged 이벤트 수신 시 즉시 재조회 가능

주의:
- QuotePoller는 SettingsWindow를 모른다.
- UI 이벤트를 직접 다루지 않는다.


--------------------------------------------------
4.6 MainWindow
--------------------------------------------------

역할:
- 주가 표시 UI
- 트레이 상주 미니 창

주요 기능:
- WatchList 변경 자동 반영
- 마우스 오버 시 투명도 1.0
- 창 위치 이동 가능
- 위치/투명도 디바운스 저장

주의:
- MainWindow는 설정 저장을 직접 결정하지 않는다.
- 설정 변경은 SettingsService를 통해서만 반영한다.


--------------------------------------------------
4.7 SettingsWindow
--------------------------------------------------

역할:
- 종목 관리 UI
- 설정 변경 인터페이스

중요 원칙:
- Config.WatchList를 직접 수정한다
- 복사본을 만들지 않는다
- Poller를 직접 호출하지 않는다

기능:
- 종목 추가 / 삭제
- 드래그로 순서 변경
- 활성 / 비활성 토글
- 투명도 설정


--------------------------------------------------
5. 설정 변경 흐름 (중요)
--------------------------------------------------

SettingsWindow
   ↓
Config.WatchList 변경
   ↓
ObservableCollection 이벤트 발생
   ↓
UI 자동 갱신
   ↓
SettingsService.NotifyChanged()
   ↓
MainWindow / QuotePoller 반응


--------------------------------------------------
6. 자주 발생하는 설계 실수
--------------------------------------------------

- WatchList를 List<T>로 교체
- WatchList를 새 컬렉션으로 대입
- SettingsWindow에서 Poller 직접 호출
- Items.Refresh() 사용
- 설정 저장과 UI 반영을 혼동


--------------------------------------------------
7. 확장 시 권장 패턴
--------------------------------------------------

- 새로운 설정 추가 → AppConfig에만 추가
- UI 옵션 → SettingsWindow에서 직접 바인딩
- 서버 기능 추가 → QuotePoller 확장
- 알림 기능 → SettingsService 이벤트 활용


--------------------------------------------------
8. 마지막 주의사항
--------------------------------------------------

이 프로젝트는 "단순함"이 가장 큰 가치이다.

구조가 복잡해지기 시작하면:
- 복사본이 생기고
- 동기화 코드가 늘어나며
- 버그가 기하급수적으로 증가한다.

의심스러울 때는:
"이 코드는 Config를 직접 수정하고 있는가?"
를 기준으로 판단할 것.
